Interviewer (AI)
Your nightly pipeline fails halfway and pages you at 3am. How is it built so that's rare, and recoverable when it happens?
Lena
Each task is idempotent and writes to a staging table before an atomic merge, so a partial run leaves no half-applied data. Tasks are partitioned by date with checkpoints, so on failure I rerun only the failed partition, not the whole job. SLAs and freshness checks alert before downstream dashboards break.
Interviewer (AI)
How do downstream teams know if the data they're reading is trustworthy that morning?
Lena
I publish a data contract per table — schema, freshness SLA, and owner — and dbt tests gate publication. If tests fail, the table isn't promoted and a status flag tells consumers it's stale rather than silently serving bad numbers.
Reviewer rationale
  • Designed for failure (staging + atomic merge + partition reruns) without being prompted.
  • Thinks past the pipeline to the consumer — data contracts and trust signals.
  • Concrete and specific; no hand-waving about 'best practices'.
