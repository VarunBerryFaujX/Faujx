What the candidate submitted
Idempotency-Key header -> upsert into idempotency table behind a unique constraint, inside the same transaction as the write; on conflict return the stored response. SELECT ... FOR UPDATE to serialize concurrent duplicates. Table-driven tests for the retry path, the concurrent-duplicate race, and a partial-failure rollback.

Interviewer (AI)
A nightly job double-posted a batch of claim adjustments to the ledger. Walk me through how you contain it and prevent recurrence.
Daniel
Contain first: stop the job, quantify the blast radius from the ledger, and reverse the duplicates with compensating entries rather than deleting rows — the audit trail has to stay intact. Then root cause: it's almost always a missing idempotency boundary or an unsafe retry. I'd make the write idempotent on a natural key with a uniqueness constraint so a replay can't double-post.
Interviewer (AI)
The job runs across several workers. How do you stop two workers grabbing the same batch?
Daniel
Claim work with a transactional lock — SELECT ... FOR UPDATE SKIP LOCKED, or a leased queue with a visibility timeout. Each row is owned by exactly one worker until it's done or the lease expires, so a crash doesn't strand or duplicate work.
Reviewer rationale
  • Reached for compensating entries and preserving the audit trail unprompted — correct instinct for a regulated ledger.
  • Tied the fix to idempotency plus a DB constraint, not a one-off patch.
  • Knew SKIP LOCKED / leased-queue ownership for concurrent workers — senior-level.
